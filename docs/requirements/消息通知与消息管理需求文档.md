# 消息通知与消息管理需求文档（PRD）

## 1. 文档信息
- 文档名称：消息通知与消息管理需求文档
- 版本：V1.0
- 日期：2026-03-06
- 适用端：移动端（snowy-mobile）+ 后端服务（snowy-plugin-biz / 相关消息服务）

## 2. 背景与目标
当前系统缺少统一的消息中心能力，用户无法高效感知审核进度、政策更新与风险预警，也缺乏消息筛选、状态管理、清理和导出能力。

本次建设目标：
1. 消息通知：集中推送审核、政策、预警三类通知。
2. 消息管理：支持按类型筛选、已读/未读区分，可删除与导出消息。

## 3. 业务范围
### 3.1 本期范围（In Scope）
- 消息中心首页与列表页。
- 三类消息统一入库与展示：
  - 审核通知（如：贷款审核、投保审核、理赔审核等）。
  - 政策通知（平台政策、补贴政策、规则更新）。
  - 预警通知（设备异常、项圈异常、业务风险预警）。
- 消息状态：未读、已读。
- 消息管理：
  - 按类型筛选。
  - 按已读/未读筛选。
  - 单条删除、批量删除（可选）。
  - 导出消息（按筛选条件导出）。

### 3.2 非本期范围（Out of Scope）
- 站外通知渠道（短信、邮件、微信模板）的深度编排。
- 消息模板可视化编辑器。
- 复杂订阅规则（用户自定义关键词订阅）。

## 4. 角色与权限
- 普通用户：查看、筛选、标记已读、删除、导出本人可见消息。
- 管理员：可查看权限范围内消息（按组织/养殖场/租户规则），可执行同等管理操作。
- 超级管理员：可查看全量（遵循数据权限框架）。

## 5. 功能需求

## 5.1 消息通知（集中推送）
### 5.1.1 分类定义
- 审核通知（AUDIT）
- 政策通知（POLICY）
- 预警通知（ALERT）

### 5.1.2 通知字段
- `id`：消息ID
- `type`：消息类型（AUDIT/POLICY/ALERT）
- `title`：标题
- `content`：正文
- `sourceBizType`：业务来源类型（如 LOAN/INS/CLAIM/DEVICE）
- `sourceBizId`：业务主键
- `farmId`：养殖场ID（可为空）
- `status`：read/unread
- `publishTime`：发布时间
- `readTime`：阅读时间
- `operator`：触发人/系统

### 5.1.3 推送规则（V1）
- 审核流程状态变更时生成审核通知。
- 政策发布后按可见范围批量生成政策通知。
- 预警事件触发时生成预警通知。
- 消息写入消息中心后，移动端消息角标实时/准实时更新。

## 5.2 消息管理
### 5.2.1 列表展示
- 默认按 `publishTime DESC` 排序。
- 每条消息显示：类型标签、标题、摘要、发布时间、已读状态。

### 5.2.2 筛选条件
- 类型筛选：全部/审核/政策/预警。
- 状态筛选：全部/未读/已读。
- 时间筛选（可选增强）：开始时间、结束时间。

### 5.2.3 已读/未读
- 点击消息详情自动标记已读。
- 支持“全部标记已读”（可选）。

### 5.2.4 删除
- 支持单条删除。
- 支持批量删除（勾选后删除）。
- 删除后前端立即更新数量及列表。

### 5.2.5 导出
- 导出当前筛选条件下消息。
- 支持字段：类型、标题、内容、发布时间、已读状态、阅读时间。
- 默认格式：`xlsx`（或 `csv`，由后端实现成本确定）。

## 6. 交互与页面需求
### 6.1 页面结构
- 消息入口（首页角标或“我的-消息”）。
- 消息列表页：
  - 顶部筛选栏（类型、状态）
  - 列表区
  - 管理操作区（删除、导出）
- 消息详情页：
  - 标题、时间、类型、正文
  - 来源业务跳转按钮（如“查看审核单”）

### 6.2 关键交互
1. 用户进入列表页，默认加载“全部+全部”。
2. 切换筛选条件后刷新列表。
3. 点击消息进入详情并自动设为已读。
4. 勾选消息后可执行删除。
5. 点击导出，下载当前筛选结果文件。

## 7. 数据模型建议
表：`biz_message_center`
- `id` varchar PK
- `tenant_id` varchar
- `user_id` varchar
- `farm_id` varchar
- `type` varchar(20)
- `title` varchar(200)
- `content` text
- `source_biz_type` varchar(50)
- `source_biz_id` varchar(64)
- `read_flag` varchar(20) default 'UNREAD'
- `publish_time` timestamp
- `read_time` timestamp null
- `delete_flag` varchar(20) default 'NOT_DELETE'
- `create_time` timestamp
- `create_user` varchar
- `update_time` timestamp
- `update_user` varchar

索引建议：
- `(user_id, read_flag, type, publish_time desc)`
- `(tenant_id, publish_time desc)`
- `(source_biz_type, source_biz_id)`

## 8. 接口需求（建议）
1. `GET /biz/message/page`
- 入参：`type`、`readFlag`、`pageNo`、`pageSize`、`startTime`、`endTime`
- 出参：分页消息列表

2. `GET /biz/message/detail?id=xxx`
- 出参：消息详情
- 动作：查询成功后自动标记已读

3. `POST /biz/message/read`
- 入参：`ids` 或 `all=true`
- 出参：处理结果

4. `POST /biz/message/delete`
- 入参：`ids`
- 出参：处理结果

5. `GET /biz/message/export`
- 入参：同列表筛选条件
- 出参：文件流

## 9. 非功能需求
- 性能：
  - 列表接口 P95 < 500ms（分页 20 条）
- 安全：
  - 严格按租户与数据权限过滤
- 可用性：
  - 导出失败需有明确错误提示
- 可观测性：
  - 关键操作日志：读取、删除、导出

## 10. 验收标准（UAT）
1. 三类消息可被正确生成并在消息中心展示。
2. 类型筛选、已读/未读筛选结果准确。
3. 点击详情后状态由未读变已读。
4. 删除后列表与计数即时更新。
5. 导出文件内容与筛选结果一致。
6. 无权限用户无法查看非授权消息。

## 11. 里程碑建议
- M1：数据模型与基础接口（2~3 天）
- M2：移动端列表/详情/筛选（2~3 天）
- M3：删除与导出能力（1~2 天）
- M4：联调与验收（1~2 天）

## 12. 风险与备注
- 风险：历史业务事件补发消息时可能产生重复，需要幂等键控制（`source_biz_type + source_biz_id + type + user_id`）。
- 备注：若需支持多终端消息同步，建议后续增加 WebSocket 或轮询角标刷新机制。
